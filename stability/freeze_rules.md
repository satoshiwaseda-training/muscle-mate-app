# stability — 緊急停止（Freeze）定義書

## 目的
KPIが規定値を超えて悪化した場合、または重大リスクが発生した場合に、
進行中の全変更・実験を即時停止し、ユーザー体験の安定を最優先で回復する。
Freezeは「組織の安全装置」であり、発動を躊躇することは禁止。

---

## Freeze レベル定義

### 🔴 FREEZE-CRITICAL（最高レベル）
**発動条件のいずれか1つを満たした時点で即時発動。手動承認不要。**

| 条件ID | 条件 | 計測ウィンドウ |
|--------|------|-------------|
| FC-001 | D7継続率が前週比 -10pt以上の悪化 | 直近7日間 |
| FC-002 | 本番エラー率が 1%超 かつ 30分以上継続 | 30分間 |
| FC-003 | 決済エラー率が 5%超 | 直近1時間 |
| FC-004 | 個人情報漏洩の可能性をlegalまたはCTOが検知 | 即時 |
| FC-005 | App Store / Google Play からの緊急勧告または削除警告 | 即時 |
| FC-006 | クラッシュ率が 1%超 かつ 15分以上継続 | 15分間 |

**FREEZE-CRITICAL 発動時の自動アクション（5分以内に全実行）:**
```
1. 進行中の全ロールアウトを Stage 1 以下にロールバック
2. 進行中の全A/Bテストを Control群（現行）に統一
3. evolution の実験開始を一時停止
4. CEO・CTO・CPO・CFOに即時通知（freezeIDをセットで送付）
5. ops_realtimeと連携して障害対応フローを開始
```

**発動条件のいずれか1つを満たした場合に発動。CTO確認後に実行（5分以内）。**

| 条件ID | 条件 | 計測ウィンドウ |
|--------|------|-------------|
| FH-001 | D1継続率が前週比 -8pt以上の悪化 | 直近3日間 |
| FH-002 | workout完了率が 55%未満に低下 | 直近24時間 |
| FH-003 | アンインストール数が前週比 +50%以上増加 | 直近24時間 |
| FH-004 | API応答時間P95が 2,000ms超 かつ 1時間以上継続 | 1時間間 |
| FH-005 | 日次AIコストが上限の 90%（45,000円）を超過 | 当日累計 |
| FH-006 | セキュリティスキャンで「高」リスク脆弱性を検知 | 即時 |

**FREEZE-HIGH 発動時のアクション（CTO確認後5分以内）:**
```

2. 当該KPIに関連するA/Bテストを一時停止
3. CEO・CTO・CPOに通知
4. CTOが原因特定を開始（30分以内に中間報告）
```

## Freeze 解除ルール（厳守）

### 解除条件（全レベル共通）
以下の**全条件**を満たすまでFreezeを解除しない：

```
条件 1: Freeze発動のトリガーKPIが閾値を連続24時間上回っていること
条件 2: 原因が特定され、文書化されていること
条件 3: 再発防止策がlearning_repositoryに記録されていること
条件 4: CTO・CPO・CEOが解除を承認していること（CRITICAL / HIGH の場合）
         CTO・CPOが解除を承認していること（MEDIUM の場合）
```

**「様子見で解除」は禁止。数値で解除する。**

### 解除フォーマット
```
【Freeze 解除通知】
Freeze ID: FRZ-[YYYYMMDD]-[連番3桁]
解除時刻: YYYY-MM-DD HH:MM
Freeze 継続時間: [X]時間[Y]分

解除理由:
  - [KPI名] が [X]%（閾値 [Y]%）を [N]時間連続で上回ったことを確認

根本原因: [具体的に1〜3文]
再発防止策: [具体的に1〜3文]
learning_repository 登録ID: LRN-[ID]

解除後に再開するロールアウト/実験:
  - [ロールアウトID]: Stage [N] から再開
  - [実験ID]: 計測を再開
```

## Freeze 中の許可アクション

Freeze 発動中も以下のアクションは実行可能：
- 本番障害の修正（凍結変更を修正する方向の変更のみ）
- ユーザーへの告知バナー表示（既存定型文のみ）
- セキュリティ修正
- KPI監視・データ収集（変更なし）

**Freeze 中に禁止するアクション:**
- 新機能のリリース
- A/Bテストの新規開始
- evolutionの仮説→実験フローの開始
- AIプロンプトの変更
- DBスキーマの変更

## Freeze とevolution / stability の関係

```
evolution（常時実験・学習ループ）
  ↓ 実験の実施申請
stability（変更バジェット + rollout_control で速度制御）
  ↓ KPI悪化を検知
freeze_rules（即時停止・安全装置）
  ↓ KPI回復確認後
stability が解除承認
  ↓
evolution がループを再開
```

**freeze はevolution を「殺す」のではなく「一時停止して回復させる」装置。**
Freeze 解除後、evolutionは停止前の状態から再開する。
