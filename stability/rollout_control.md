# stability — 段階ロールアウト定義書

## 目的
全ユーザーへの一斉リリースを禁止し、
段階的にユーザーへ変更を展開することで、
KPI悪化を最小ユーザー数で検知・停止できる体制を作る。

---

## ロールアウト段階の定義（全変更に適用）

### 標準ロールアウト（CPが3以上の変更に必須）

```
Stage 0: 内部テスト（0%・社内ユーザーのみ）
  期間: 1日
  通過基準: エラー率 0%・クラッシュなし
  失敗時: 即時停止・修正後にStage 0から再開始
    ↓
Stage 1: カナリアリリース（5%・ランダム抽出）
  期間: 24時間以上
  通過基準: エラー率 < 0.5%・対象KPI悪化なし（±2%以内）
  失敗時: 即時ロールバック・freeze_rules確認
    ↓
Stage 2: 限定公開（20%・ランダム抽出）
  期間: 48時間以上
  通過基準: エラー率 < 0.3%・対象KPI +1pt以上 または 悪化なし
  失敗時: 即時ロールバック・freeze_rules確認
    ↓
Stage 3: 拡大公開（50%・ランダム抽出）
  期間: 48時間以上
  通過基準: エラー率 < 0.2%・対象KPI +1pt以上
  失敗時: 即時ロールバック
    ↓
Stage 4: 全体公開（100%）
  全段階クリア後にstabilityが承認
```

### 軽量ロールアウト（CP 1〜2の変更・テキスト系）

```
Stage 0: 内部確認（1日）
    ↓
Stage 1: 20%公開（24時間）
    ↓
Stage 2: 100%公開
```

### バジェット外緊急修正（Criticalアラート対応）

```
Stage 0: 修正確認（30分以内）
    ↓
Stage 1: 100%公開（段階省略）
  ※ 事後24時間以内にポストモーテム作成
```

---

## 各ステージの通過判定フロー

```
ステージ期間終了
  ↓
stabilityが以下を確認:
  [1] エラー率が基準以内か（CTOのログから取得）
  [2] 対象KPIの変化（analyticsから取得）
  [3] ガードレール指標の変化（D7継続率・workout完了率）
  ↓
全条件クリア → 次のステージへ承認
いずれか失敗 → 即時ロールバック + freeze_rules.mdを参照
```

---

## ガードレール指標（全ステージ共通）

以下の指標が悪化した場合、ステージ判定に関わらず即時停止する：

| ガードレール指標 | 停止閾値 | 計測ウィンドウ |
|--------------|---------|-------------|
| D1継続率 | 前週比 -5pt以上の悪化 | 24時間 |
| workout完了率 | 60%未満に低下 | 12時間 |
| API エラー率 | 0.5%超 | 1時間 |
| クラッシュ率 | 0.1%超 | 1時間 |
| アンインストール数 | 前日比 +30%以上 | 24時間 |

## 同時ロールアウトの制限

**同時に進行できるロールアウトの上限: 2件**

3件以上を同時進行しようとした場合:
- KPI悪化が発生した際に「どの変更が原因か」を特定できなくなる
- stabilityが3件目の申請を差し戻す
- CPO・CTOが優先順位を決めて2件以内に絞る

**例外:** バジェット外緊急修正は上限にカウントしない

---

## ロールアウト失敗時のコミュニケーション

```
Stage 1以降で失敗が発生した場合:
  1. stabilityがCTO・CPO・CEOに即時通知
     通知内容: 「ROL-[ID] Stage[N]で失敗。[ガードレール指標名]が[X]→[Y]に悪化。
               即時ロールバックを実行します。」
  2. CTOがロールバックを5分以内に実行
  3. stabilityがfreeze_rules.mdの確認を実施
  4. 失敗原因をCTOが24時間以内に特定・文書化
  5. evolutionのlearning_repositoryに失敗記録を追加
```
