# stability — 変更バジェット定義書

## 目的
週あたりの変更量に上限を設け、
複数の変更が同時にユーザー体験に影響することを防ぐ。
変更が多すぎると「何が効いたか / 何が壊したか」が特定できなくなる。

---

## 変更バジェットの単位：変更ポイント（CP: Change Point）

全ての変更は以下のテーブルに基づいてCPを消費する。
週あたり消費上限は **20CP**。
上限を超える変更はCEO承認なしに実行できない。

---

## CPコスト定義テーブル

### フロントエンド変更
| 変更種別 | CP消費 | 例 |
|---------|-------|---|
| テキスト・コピーの変更 | 1CP | ボタン文言・通知文言の変更 |
| 色・アイコン・余白の変更 | 1CP | ブランドカラーの調整 |
| 既存画面のレイアウト変更 | 3CP | ホーム画面のカード配置変更 |
| 新規画面の追加 | 5CP | 報酬画面の新設 |
| オンボーディングフローの変更 | 8CP | ステップ数・順序の変更 |
| ナビゲーション構造の変更 | 10CP | タブバーの項目変更 |

### バックエンド変更
| 変更種別 | CP消費 | 例 |
|---------|-------|---|
| AIプロンプトの調整 | 2CP | 指示文の変更・追加 |
| APIエンドポイントのロジック変更 | 3CP | スコア計算式の変更 |
| DBスキーマ変更（後方互換あり） | 5CP | カラム追加 |
| DBスキーマ変更（後方互換なし） | 10CP | テーブル再設計 |
| 外部API統合（新規） | 8CP | 新規SaaS連携 |
| 認証・権限ロジックの変更 | 10CP | ログインフロー変更 |

### インフラ変更
| 変更種別 | CP消費 | 例 |
|---------|-------|---|
| 環境変数の追加 | 1CP | APIキーの追加 |
| Vercel設定変更 | 2CP | リダイレクト設定 |
| Supabase RLSポリシー変更 | 5CP | アクセス制御ルール変更 |
| インフラ構成の変更 | 8CP | エッジ関数の追加 |

### A/Bテスト・実験
| 変更種別 | CP消費 | 例 |
|---------|-------|---|
| テキストのA/Bテスト開始 | 1CP | コピーの2パターン検証 |
| UIのA/Bテスト開始 | 3CP | レイアウト変更の実験 |
| 機能のA/Bテスト開始 | 5CP | 新機能の段階公開 |
| 複数変数の同時A/Bテスト | 禁止 | — |

## 週次バジェット運用ルール

### リセットタイミング
- 毎週月曜 00:00 に20CPにリセット
- リセット時に前週の消費レポートをCEO・全部門に自動配布

### 消費の申請フロー
```
変更を実施する前に stability に「変更申請」を提出:
  - 変更種別（上記テーブルから選択）
  - 消費CP数
  - 変更理由
  - 対象KPI

stabilityがバジェット残量を確認:
  残量あり → 承認・バジェットを消費として記録
  残量不足 → 差し戻し（次週繰越 or 変更スコープ縮小を提案）
```

### バジェット消費の優先順位
同週に複数の変更申請が競合する場合、以下の順で優先する：

1. **freeze解除のための修正**（バジェット外・無制限）
2. **ops_realtimeからのCriticalアラート対応**（バジェット外・無制限）
3. **D30継続率への影響「高」の変更**
4. **D7継続率への影響「高」の変更**
5. **コスト削減施策**
6. **その他（UI改善・新機能など）**

### バジェット外（無制限）の変更
以下はCPを消費しない（ただし実行後に記録必須）：
- 本番障害の緊急修正（ops_realtimeのCritical対応）
- セキュリティ脆弱性の修正（legal / CTOの判断）
- freeze解除のためのロールバック

```

■ 今週の消費: [X]CP / 20CP
■ 残量: [Y]CP
■ 来週への繰越: なし（毎週リセット）

■ 消費内訳:
  - [変更種別]: [X]CP（担当: [部門]・変更理由: ○○）
  - [変更種別]: [X]CP（担当: [部門]・変更理由: ○○）

■ 差し戻した申請: [N]件
  - [変更種別]: 残量不足のため来週に繰越

  - [内容]: [理由（Criticalアラート対応など）]

■ 来週の申請予定（先着順）:
  - [変更種別]: [X]CP（申請部門: [部門]）
```

## バジェット増枠申請

以下の条件を満たす場合のみ、週次上限（20CP）の増枠を申請できる：

**増枠申請条件:**
- 競合の大型アップデートへの緊急対応
- KPI急落（D7継続率 -10pt以上）への緊急改善
- リリース済みバグの修正が複数箇所にわたる場合

**増枠申請フォーマット:**
```
申請部門: [部門名]
追加申請CP数: [X]CP
理由: [具体的な状況]
影響するKPI: [KPI名 + 現状値]
承認者: CFO + CEO（両名の承認が必要）
```
