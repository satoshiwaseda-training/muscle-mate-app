# Muscle Mate — 組織連携プロトコル

## 目的
各部門（CEO / CPO / CTO / CMO / CFO）および各エージェント（planner / executor / critic / optimizer）が独立して動くことを禁止し、
定義された順序と入出力ルールに従って連携することで、
意図しない品質劣化・KPI悪化・倫理違反を防ぐ。

---

## 1. 開発フロー（全変更に適用）

### 標準フロー（部門レベル）
```
CPO（仕様定義）
  ↓
CTO（実装）
  ↓
CMO（表現・文言調整）
  ↓
CFO（リスク・規約チェック）
  ↓
CEO（最終判断・承認）
```

### エージェントフロー（AI実行レベル）
CTOフェーズの内部は以下のエージェントループで処理する：

```
CPO（仕様定義）
  ↓
planner_agent（タスク分解・実行計画）
  ↓
executor_agent（実装・変更の実行）
  ↓
critic_agent（品質・KPI・倫理の評価）
  ↓
optimizer_agent（コスト・品質の最適化）
  ↓
CMO（表現・文言調整）
  ↓
CFO（リスク・規約・AI法チェック）
  ↓
CEO（最終判断・承認）
```

**critic_agentが「差し戻し」を出した場合:**
```
critic_agent（差し戻し）
  → executor_agent（再実行）
  → critic_agent（再評価）
  ※ 差し戻しは最大2回まで。3回目はCTOにエスカレーション
```

### フロー適用範囲
- 新機能の開発
- 既存機能の仕様変更
- UIコピー・通知文言の変更
- 料金・課金フローの変更
- AIの応答ロジックの変更
- AIモデルのルーティング変更

### フロー省略が許可される範囲
| 変更種別 | 省略可能なステップ | 条件 |
|---------|-----------------|------|
| バグ修正（UI変更なし） | CMO・CFO | CTOが影響範囲を文書化すること |
| タイポ・誤字修正 | すべて省略可 | CPOが事後報告すること |
| インフラ設定変更 | CMO・エージェントフロー | CTOがリスク評価を自己完結すること |
| キャッシュTTL変更のみ | エージェントフロー・CMO | CTOとCFOが合同承認すること |

## 2. 部門間の入出力ルール

### CPO → CTO（仕様の受け渡し）

**CPOが渡すもの（必須）:**
- ユーザーストーリー: 「[ユーザー種別]として、[目的]のために、[機能]が欲しい」の形式
- UI仕様: Wireframeまたはスクリーンレイアウトの文書
- 必要ロジック: 入力・処理・出力の定義（擬似コードで可）
- ユーザー体験目標: behavior_model.mdのどのステップに作用するか
- 受け入れ基準: 「〇〇ができれば完了」を3項目以上

**CTOが受け取り拒否できる条件:**
- UI仕様が存在しない場合
- 受け入れ基準が未定義の場合
- 継続率への影響度が「高/中/低」のいずれかで記載されていない場合

---

### CTO → CMO（実装結果の受け渡し）

**CTOが渡すもの（必須）:**
- 実装結果のサマリー: 何が動くようになったかを1〜3文で記述
- 表示テキストの一覧: ボタン・ラベル・通知・エラーメッセージのすべて
- ロジック説明: ユーザーに見える振る舞いの説明（技術詳細は不要）
- 変更前後の差分: スクリーンショットまたはテキスト差分

**CMOが受け取り拒否できる条件:**
- 表示テキスト一覧が不完全な場合
- ユーザーに見える振る舞いの説明がない場合

---

### CMO → CFO（表現内容の受け渡し）

**CMOが渡すもの（必須）:**
- 最終表現案: 全テキスト・コピー・通知文言の確定版
- ナッジ設計の説明: どのような行動変容を意図しているか
- ターゲットユーザーの定義: どのセグメントに向けた表現か
- 倫理チェック自己申告: ethics_policy.mdの各項目に対して違反なしと確認済みであること

**CFOが受け取り拒否できる条件:**
- 倫理チェック自己申告がない場合
- ナッジ設計の意図が不明な場合

---

### CFO → CEO（最終承認申請）

**CFOが渡すもの（必須）:**
- リスク評価: 法的・財務的・倫理的リスクの3軸で評価（高/中/低/なし）
- コスト影響: インフラコスト・APIコストへの影響試算
- 規約適合確認: 利用規約・プライバシーポリシーへの適合性
- 承認推奨: 承認・条件付き承認・却下のどれかを明記し、理由を添える

**CEOが最終承認・却下を行う基準:**
- リスクが「高」の項目が1つ以上ある場合: 条件付き承認または却下
- KPI影響の記述がない場合: 差し戻し
- behavior_model.mdのフローに逆行する場合: 即時却下

## 3. 禁止事項

| 禁止行為 | 違反した場合の対応 |
|---------|-----------------|
| 部門をまたいで他部門の担当範囲を勝手に変更する | 変更を即時ロールバックし、フローを最初からやり直す |
| UIロジックとバックエンドロジックを同一PRで同時変更する | PRを分割して再提出する |
| KPIへの影響記述なしに変更を申請する | CFOが受理せず差し戻す |
| フローを省略して直接CEOに承認を求める | CEOは受理せず、省略されたステップに差し戻す |
| 「急いでいる」を理由にフローを飛ばす | 緊急対応フロー（下記）を使うこと |

---

## 4. 強制ルール

### 全変更に必須の記述項目
すべての変更申請・PR・Issueには以下を必ず含めること：

```
## 変更の目的
（何を解決するか、1〜2文で）

## KPIへの影響
（どのKPIに・どの程度・どの方向に影響するか）
例: D7継続率 +2pt（予測）、APIコスト 変化なし

## behavior_model.mdとの対応
（理解/納得/行動/報酬/継続のどのステップに作用するか）

## 影響範囲
（変更されるファイル・DB・API・UI要素の一覧）
```

### 理由なき変更の禁止
上記4項目が揃っていない変更は、どの部門も受理してはならない。
「とりあえず試す」「なんとなく改善」は変更理由として認めない。

## 5. 緊急対応フロー（本番障害時）

通常フローが間に合わない本番障害の場合のみ、以下の短縮フローを使用する：

```
CTO（障害対応の即時実施）
  ↓ 同時並行で
CEO（口頭またはチャットで承認）
  ↓ 対応後24時間以内に
CPO / CMO / CFO（事後レビュー・文書化）
```

**緊急対応フロー適用条件（すべてを満たすこと）:**
- 本番環境でユーザーに影響が出ている
- 通常フローでは対応完了まで2時間以上かかる
- CTOがCEOに状況を口頭・チャットで共有済みである

---

## 5. 強制前進ルール（Critical）

すべての部門は以下のルールに従い、常に前進し続けること。

### 1. 不完全でも次工程に渡す
- 完璧な仕様を待たない
- 70%完成の状態で次の部門に渡して良い
- 渡す際に「未確定箇所」を明示すること

### 2. 不明点は仮定して進める
- 情報不足を理由に停止しない
- 仮定を明記した上で進行する
- 仮定の記述フォーマット: `【仮定】○○と想定して進める。確認が取れ次第修正する。`

### 3. 各部門の最大停止時間
| 部門 | 最大停止時間 | 超過した場合 |
|------|------------|------------|
| CPO | 5分 | 仮定を置いて次工程に渡す |
| CTO | 10分 | 仮定を置いて次工程に渡す |
| CMO | 5分 | 仮定を置いて次工程に渡す |
| CFO | 5分 | 仮定を置いて次工程に渡す |

### 4. 「停止」はCEO承認が必要
- 最大停止時間を超えての作業中断はCEO承認が必要
- CEO承認なしの停止は違反とみなす
- 承認申請フォーマット: `【停止申請】部門名 / 理由 / 再開見込み時刻`

### 5. 出力が空は禁止
- 成果物ゼロでの引き渡しは禁止
- 最低限の成果物として「現状の仮定リスト」または「判明している制約の一覧」を出すこと
- 「分からない」は出力ではない。「○○が不明なので△△と仮定した」が出力である

## 6. v3.0 追加部門の連携ルール

v3.0で追加された5部門は、既存フローに以下の形で組み込まれる。

### analytics の連携位置
```
analytics は全部門に数値を供給する「横断インフラ」として機能する。
フロー上のステップではなく、各部門が参照するデータソース。

参照タイミング（必須）:
- growth: 週次グロースサイクルの月曜（週次レポート受け取り）
- CPO: 機能優先度決定時（KPI影響の根拠として）
- CMO: 施策効果測定時
- monetization: LTV・解約率レポートの確認
```

### growth の連携位置
```
CPO（機能仕様）← growth（改善仮説を提供）
CMO（獲得施策）← growth（リファラル・オンボーディング仕様を提供）
analytics → growth（週次KPIレポートを供給）
```

### monetization の連携位置
```
通常フロー内では CPO の前段に位置する：
monetization（収益設計）→ CPO（機能仕様化）→ 通常フロー

ペイウォール・課金フローの変更時は必須ステップ：
monetization → CPO → CTO → CMO → CFO → legal → CEO
```

### infra_cost の連携位置
```
infra_cost は CTO と CFO の間に位置するコスト検証レイヤー：
optimizer_agent（コスト試算）→ infra_cost（実績との照合・承認）→ CFO

月次コストアラート時:
infra_cost → CTO（技術対応）+ CFO（予算対応）同時通知
```

### legal の連携位置
```
legal は CFO の直前に位置する法的ゲートキーパー：

通常フロー（法的影響のある変更時）:
optimizer_agent → CMO → legal（法的リスク評価）→ CFO → CEO

法的リスク「高」判定時:
legal が即時差し止め → CEO へ報告 → 解除条件の充足まで前進禁止
```

### v3.0 フルフロー（全部門統合）
```
analytics（数値供給）
  ↓ ← growth（改善仮説）
CPO（仕様定義）
  ↓
planner_agent → executor_agent → critic_agent → optimizer_agent
  ↓
CMO（表現調整）
  ↓
legal（法的リスク評価）※ 法的影響がある変更のみ
  ↓
CFO + infra_cost（財務・コストチェック）
  ↓
CEO（最終承認）
```

## 7. 意思決定のエスカレーション基準

| 状況 | エスカレーション先 |
|------|----------------|
| 部門間の意見が対立して決まらない | CEO |
| 倫理ポリシーに抵触する可能性がある | CFO → CEO |
| KPIに重大な悪影響が予想される | CPO + analytics → CEO |
| セキュリティリスクが発見された | CTO → CEO（即時） |
| 予算を30%以上超過する見込み | infra_cost → CFO → CEO |
| 法的リスク「高」が検出された | legal → CEO（即時差し止め） |
| D30継続率が目標を3週連続で下回る | growth + analytics → CEO |
| AIコストが月次上限の80%を超えた | infra_cost → CTO + CFO（同時）|
| alert_rules.mdのCriticalが発報した | ops_realtime が2分以内に自動対応を開始 |
| 自動対応で30分以内に解決しない場合 | ops_realtime → CEO（エスカレーション通知） |

### stability の連携位置
```
stabilityはevolutionとops_realtimeの「間」に位置する速度制御レイヤー。

変更が発生するたびに:
  申請部門 → stability（変更バジェット確認）→ rollout_control（段階展開）
                                               ↓ KPI悪化検知
                                          freeze_rules（即時停止）
                                               ↓ KPI回復確認後
                                          stability が解除承認
                                               ↓
                                          evolution がループ再開

stabilityが超えられない原則:
  - 週次変更バジェット（20CP）を超える変更はCEO承認なしに実行不可
  - freeze条件（17条件）に達した変更は即時停止・解除条件充足まで再開不可
  - evolutionの仮説はstabilityのバジェット内に収まるよう「実装サイズを縮小」して実施
    （evolutionを「停止」するのではなく「速度調節」する）

freeze発動時の最優先原則:
  CRITICAL → 5分以内に自動停止（手動承認不要）
  HIGH     → CTO確認後5分以内
  MEDIUM   → CPO+CTO協議後30分以内
```

### evolution の連携位置
```
evolutionはstrategy_loopで組織全体の意思決定サイクルを駆動する「戦略エンジン」。

入力（参照するデータ）:
  analytics        → 週次・月次KPIレポート
  market_feedback  → ストアレビュー・競合情報・インタビュー結果
  learning_repository → 過去の実験結果・失敗知見

出力（アクションを起こす先）:
  CPO     ← 成功施策の本番適用提案
  CMO     ← コミュニケーション変更仮説の実験依頼
  growth  ← 継続率改善仮説の共有
  analytics ← 実験計測イベントの追加依頼

evolutionが駆動するループ:
  仮説生成（hypothesis_engine）
    → 実験設計・承認（auto_experiment）
      → 実装（CPO → 通常開発フロー）
        → 計測（analytics）
          → 判定・記録（learning_repository）
            → 次の仮説生成 ← 無限ループ
```

### ops_realtime の連携位置
```
ops_realtimeは通常の開発フロー外に存在する「常時稼働の監視レイヤー」。
開発フローを止める権限は持たないが、緊急時にCEOへ直接報告できる唯一の部門。

異常検知時の通知先:
  Critical → CEO + 該当部門（CTO/CPO/CFO）に同時通知
  High     → 該当部門のみに通知
  Medium   → 該当部門のみに通知（CEOには週次サマリーで報告）

自動実行できるアクション（権限委譲済み）:
  - Vercelロールバック（エラー率1%超時）
  - AIフォールバックモード切り替え（GeminiエラーRate 5%超時）
  - サーバースケール（コスト上限内）
  - IPブロック（認証失敗20回超時）
  - 定型バナー表示
```
