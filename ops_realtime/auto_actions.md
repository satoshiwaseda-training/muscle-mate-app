# ops_realtime — 自動対応フロー定義書

## 原則
- 各アラートIDに対する対応は本ファイルに定義された通りに実行する
- 定義にないアクションは実行しない（CEOにエスカレーション）
- アクションは冪等に設計する（同じアクションを2回実行しても副作用が出ないこと）
- 全アクション実行後にログを記録する（記録なしのアクションは未実行とみなす）

---

## 1. 継続率 / エンゲージメント系 自動対応

### ALT-001 / ALT-002 / ALT-003（継続率 Critical）
**発報から2分以内に実行:**
```
Step 1: CEO・CPO・growthに即時通知
  通知内容: 「[ALT-00X] [KPI名] が [実測値]% に低下（閾値: [閾値]%）。自動対応を開始します。」

Step 2: analyticsに直近48時間の継続率コホートレポートを即時生成を要求

Step 3: growthに「継続率緊急対応」フラグを立てる
  → growthは通知受信後5分以内に原因仮説を1件提出すること

Step 4（30分後に改善が見られない場合）:
  → CEOにエスカレーション: 緊急ミーティングのアジェンダを生成して送付
```

## 2. インフラ / パフォーマンス系 自動対応

### ALT-011（API P95 > 1,000ms / Critical）
**発報から2分以内に実行:**
```
Step 1: CTO・infra_costに即時通知
Step 2: 自動でAPIヘルスチェックエンドポイントを叩き、結果をログに記録
Step 3: Supabase接続状態を確認（ALT-015と同時発報の場合はDB障害フローへ）
Step 4: 問題が継続する場合（5分後）→ Vercelのエッジキャッシュをパージ
Step 5: 解決しない場合（15分後）→ CEOに本番障害として報告・緊急対応フロー起動
```

---

### ALT-013（エラー率 > 1% / Critical）
**発報から2分以内に実行:**
```
Step 1: CTO・CEOに即時通知
Step 2: エラーの種類を自動分類（4xx / 5xx / タイムアウト）してログに記録
Step 3: 5xxエラーが50%以上 → CTOに「直近デプロイのロールバック」を通知
Step 4: ロールバック実施 → エラー率の回復を5分間モニタリング
Step 5: 回復しない場合 → CEOへ本番障害エスカレーション
```

---

### ALT-015（Supabase DB接続エラー / Critical）
**発報から2分以内に実行:**
```
Step 1: CTO・CEO・infra_costに即時通知
Step 2: Supabaseステータスページ（status.supabase.com）の状態を自動取得してログに記録
Step 3: Supabase側障害の場合 → フロントエンドに「メンテナンスバナー」を表示（CPOに承認要求）
Step 4: 当社起因の場合 → CTOが接続プールの再起動を実施
Step 5: 15分以内に解決しない場合 → CEOエスカレーション
```

---

### ALT-016（Vercelデプロイ失敗）
**発報から5分以内に実行:**
```
Step 1: CTOに即時通知（デプロイログのURLをセットで送付）
Step 2: 直前の成功デプロイへのロールバックをCTOに提案
Step 3: 本番に影響が出ている場合 → ALT-013の対応フローと並列実行
Step 4: ロールバック後に原因分析レポートをCTOが24時間以内に提出
```

---

### ALT-017 / ALT-018（CPU / メモリ高負荷）
**発報から5分以内に実行:**
```
Step 1: CTO・infra_costに通知
Step 2: 高負荷プロセスを自動特定してログに記録
Step 3（ALT-018 Memory Critical）: 自動でサーバーの水平スケールを実行（infra_cost予算範囲内）
Step 4: スケール後15分以内に正常化しない場合 → CEOエスカレーション
Step 5: コスト影響をinfra_costがCFOに即時報告
```

## 3. AI API系 自動対応

### ALT-021（Gemini APIエラー率 > 5% / Critical）
**発報から2分以内に実行:**
```
Step 1: CTO・infra_costに即時通知
Step 2: Gemini APIステータスを自動取得してログに記録
Step 3: AI依存機能（メニュー提案・フィードバック）を「静的フォールバックモード」に切り替え
  フォールバック内容: 「現在AIが一時的に利用できません。基本メニューを表示しています。」
Step 4: ユーザー向けバナーをCPO承認なしで即時表示（定型文使用）
Step 5: APIが回復後 → フォールバックモードを自動解除・ログに記録
```

**発報から5分以内に実行:**

```
Step 2: 過去2時間のAPIコール詳細をinfra_costが取得・分析
Step 3: 異常なAPIコールパターンがある場合 → CTOがレート制限を一時強化

[ALT-024: 3,000円超 / Critical]

Step 2: AIリクエストのレート制限を自動で50%に引き下げ（CTO権限委譲済み）
Step 3: ユーザー影響が出る場合 → CPOに告知バナーの即時表示を要求
Step 4: optimizer_agentにコスト削減タスクを即時割り当て

```

```
Step 4: 解約フロー正常 → monetizationが解約防止オファーの内容を緊急レビュー
```

## 5. セキュリティ系 自動対応

### ALT-041（認証失敗 同一IP 20回以上 / Critical）
**発報から2分以内に実行:**
```
Step 1: CTO・CEOに即時通知（対象IPアドレスをログに記録）
Step 2: 対象IPからのリクエストを自動ブロック（30分間・CTO権限委譲済み）
Step 3: ブロック後も別IPから継続する場合 → legalに報告・外部攻撃として記録
Step 4: 誤検知（社内IPなど）の場合 → CTOが即時解除・誤検知ログに記録
```

---

### ALT-042（同一ユーザー APIレート超過 / Critical）
**発報から2分以内に実行:**
```
Step 1: CTOに即時通知
Step 2: 対象ユーザーアカウントのAPIアクセスを自動制限（1分あたり10回に引き下げ）
Step 3: 制限後も異常が継続する場合 → アカウントを一時停止・ユーザーにメール通知
Step 4: legalに不正利用ログとして記録・規約違反として対応を協議
```

---

### ALT-044（個人情報大量アクセス / Critical）
**発報から2分以内に実行:**
```
Step 1: CTO・CEO・legalに即時通知
Step 2: 対象の読み取りAPIエンドポイントへのアクセスを自動遮断
Step 3: アクセスログを全件保全（削除・上書き禁止フラグを設定）
Step 4: legalが個人情報漏洩リスクを評価（1時間以内）
Step 5: 漏洩の可能性がある場合 → 個人情報保護委員会への報告フローを開始
```

## エスカレーション共通フロー

自動対応で解決しない場合の共通手順：

```
自動対応開始から30分以内に正常化しない
  ↓
CEOに「エスカレーション通知」を送信:
  - アラートID
  - 発報時刻
  - 実測値 vs 閾値
  - 実行済みアクション一覧
  - 現在の状態
  - 推奨される次のアクション（1件）
  ↓
CEOが判断・指示
  ↓
指示に従いops_realtimeが追加アクションを実行
  ↓
解決後: ポストモーテム作成依頼（CTO / CPO / 関連部門）
```

## 自動アクション権限委譲一覧

以下のアクションはCEO・CTOの事前権限委譲により、ops_realtimeが承認なしで実行できる：

| アクション | 委譲元 | 条件 |
|-----------|--------|------|
| Vercel直前デプロイへのロールバック | CTO | エラー率1%超の場合のみ |
| サーバーの水平スケール | CTO / CFO | 月次コスト上限の範囲内 |
| AIレート制限を50%に引き下げ | CTO | 日次AIコスト3,000円超の場合のみ |
| 同一IPの30分ブロック | CTO | 認証失敗20回以上の場合のみ |
| AIフォールバックモードへの切り替え | CPO | GeminiエラーRate 5%超の場合のみ |
| 定型バナーの表示（定義済み文言のみ） | CPO | 各アラートフロー内で明示された場合のみ |

**上記以外のアクションは必ずCEO・CTO・CPOのいずれかの承認を得ること。**
